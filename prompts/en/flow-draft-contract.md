## Draft format contract (shared by generation and repair)

`flow.semantic.json`:
- Must be a JSON object with only top-level `title` and `goals`; do not write `source`, `schemaVersion`, `status`, `currentGoal`, `parallelRun`, `checks`, or other runtime fields.
- The `goals` array order is the execution order. Each item only needs `title`, `role`, `file`, and optional `dependsOn` / `writeScope`. Do not write `index`; the extension recalculates 0-based indexes from order.
- `dependsOn` is optional. Its value is an array of 0-based indexes of earlier Goals; when omitted, it defaults to depending on the previous Goal; write `[]` when there are explicitly no prerequisites.
- `writeScope` is optional. It only accepts `**` or relative directory globs ending in `/**` (e.g. `src/api/**`); exact files and other glob syntax are forbidden. If uncertain, omit it so the scheduler conservatively serializes the Goal.
- Every Goal except final acceptance uses role `normal`; `final_acceptance` may appear at most once and only as the last Goal; do not use `implementation`.
- Each `file` must be a relative path inside the current Flow directory, and the referenced Goal markdown file must exist; Chinese titles use `G<N>-goal.md`, English titles may slug (e.g. `G1-login-ui.md`).

Minimal `flow.semantic.json` skeleton with optional parallel fields:

```json
{
  "title": "Task title",
  "goals": [
    { "title": "First Goal", "role": "normal", "file": "G1-goal.md", "dependsOn": [], "writeScope": ["src/api/**"] },
    { "title": "Final acceptance", "role": "final_acceptance", "file": "G2-final-acceptance.md", "dependsOn": [0] }
  ]
}
```

Goal markdown:
- Each Goal must be small enough to complete in its own Goal session; each file must contain the seven sections `Objective / Scope / Steps / Success Criteria / Verification / Notes / Handoff`; include the `Handoff` heading, content may be empty.
- `Success Criteria` must be ordinary bullets, not checkboxes; write completion status and evidence in `Verification` / `Handoff`, not in `Success Criteria`.
- `Success Criteria` must be bounded: write a finite, enumerable verification set (specific files, commands, scenarios); no universally quantified open-ended guarantees (such as "no bypass exists" or "any input will..."); criteria must not contradict each other or the Scope.
- `Steps` and `Verification` must use checkboxes, initially only `[ ]`; `Verification` needs an objectively verifiable command or explicit manual verification step (e.g. `- [ ] \`npm test -- --testPathPattern=auth\` exit 0`; avoid `- [ ] check that the feature works` which cannot be objectively judged).
- `Steps` are runtime todo items, not a mechanical list of tiny tasks or vague phases. Prefer 2–10 user-understandable milestones per Goal; smaller tasks may use fewer; beyond 10, split the Goal or merge overly fine steps. Each item must be independently actionable, updatable when complete, and able to provide completion evidence in `Verification` / `Handoff`; avoid big steps like "implement the feature / finish development / final check" that can only complete at the end.
- Write each Step as `- [ ] **Short title**: technical detail`; short title <= 20 Chinese chars or concise English, user-readable; detail can be technical.
  (e.g. `- [ ] **Token verification works**: implement verifyToken(token) in auth.ts, handle TokenExpiredError and invalid signature, return parsed payload`)

Final acceptance:
- An optional close-out Goal, not generated by default: a serial Flow puts global verification and docs / AGENTS.md close-out into the last `normal` Goal's Steps; generate it only when parallel batches exist or no single Goal can cover the whole integration surface.
- When generated it must be the last Goal and at most 1: the filename uses the actual sequence plus `final-acceptance` (e.g. `G3-final-acceptance.md`). It must read all Handoffs, review `criteriaChanged`, run global verification, check whether docs / AGENTS.md need updates, and close out; its Steps must cover all prior Goals' deliverables.

Boundaries and close-out:
- Only write `flow.semantic.json` and `G<N>-*.md` inside `{{flowPath}}`; do not create, handwrite, or modify canonical `flow.json` / runtime state fields (`parallelRun`, `checks`, etc.). The extension assembles and validates the complete Flow state. The Flow directory has already been allocated by the extension as bare `F<N>`; do not create another Flow directory and do not use `F<N>-slug`.
- Do not write or test `flow.html`; the extension renders it after validation with its built-in renderer. Do not copy HTML report templates or candidate structures into Goal files.
- Output language must use current language: `{{language}}`. Use English for `en`; use Chinese for `zh`.
- After writing `flow.semantic.json` and Goal markdown files, stop. The extension will assemble and run structural validation (`{{validateCommand}} {{flowPath}}`). Do not replace real tool output with "self-check", and do not manually simulate extension validation.
- If you need to write the user's next step after completion, use `/flow go F<N>`; if stopping is needed, use `/flow stop F<N>`. Do not invent another control entry.
